## Artifact Description

The artifact selected for this enhancement is the Travlr Getaways full-stack web application, originally developed during CS 465 in early 2026. By the time of this enhancement, the application had already been revised twice, since its backend was rewritten from Node.js and Express to Python and FastAPI in the first enhancement, and its trip listing endpoint gained server-side filtering, sorting, and result limiting in the second. The component addressed here is the data persistence layer beneath that backend, which until this point had continued to rely on MongoDB. In its original form, the application stored its data as two independent document collections, one for trips and one for users, with no enforced relationship between them and only minimal constraints on the shape of the data each collection held. Although that arrangement was functional, it left the responsibility for data integrity almost entirely with the application code rather than with the database itself.

## Justification for Inclusion

This artifact was selected for the databases category because migrating a document store to a normalized relational database is a direct and self-contained demonstration of the skills the category is meant to show. The enhancement replaced MongoDB with PostgreSQL, designed a normalized relational schema, and rewrote the data-access layer from the document driver to parameterized SQL executed through an asynchronous PostgreSQL client. The component that demonstrates database skill most clearly is the schema itself, which introduced three tables, namely users, trips, and a new bookings table that finally expresses the relationship the original application had never implemented despite being a travel booking application. That bookings table, with its foreign keys referencing both users and trips, is where normalization and referential integrity genuinely reside, since the two original collections had no means of referring to one another at all.

The enhancement also moved the responsibility for data integrity out of the application and into the database, which is the central value of the relational model. Where the document store had allowed a price to be stored as a string and had let duplicate trip codes accumulate over time, the relational schema enforces an exact numeric type for currency, a uniqueness constraint on the trip code, a check constraint on the number of travelers in a booking, and a not-null constraint on every required field. The query rewrite contributes to this as well, because translating the filtering and sorting from the second enhancement into SQL required treating every user-supplied value as a bound parameter and validating the requested sort field against a fixed whitelist, both of which close off the possibility of SQL injection in a way that the document interface had never required me to think about.

## Course Outcome Alignment

This enhancement was planned to address Course Outcome 5: developing a security mindset that anticipates adversarial exploits in order to mitigate design flaws and ensure the privacy and integrity of data and resources. The outcome has been met, since the migration enforces data integrity at the database level through typed columns and constraints rather than trusting the application to enforce it after the fact, and the rewritten queries resist injection by treating user input as bound parameters and validating the requested sort field against a fixed whitelist rather than leaving that input unchecked. The schema is the clearest expression of this, because its uniqueness, type, check, and not-null constraints move the guarantees about the data into the database itself, where a careless code path cannot quietly bypass them.

No updates to the outcome-coverage plan are necessary at this stage. With this enhancement complete, the three enhancements together address the algorithms and data structures, software engineering, and database categories as they were proposed in Module One, and the migration delivers on its stated objective without expanding beyond the scope that was approved.

## Reflection on the Enhancement Process

The process of completing this enhancement was instructive less for the volume of code it touched than for how quickly it exposed problems that the document store had quietly tolerated. The relational schema refused the very first attempt to load the data, because the trips collection contained duplicate codes that the original Mongoose schema had indexed but never marked as unique, which meant I had to clean the source data before it would fit a model that actually enforced uniqueness. A related issue appeared with the start date, which had been stored as a true date in the seeded records but as a string in any trip created through the API, so the migration had to normalize both representations before they would fit a single typed column. Both of these reinforced a lesson that is easy to state and easy to forget, which is that a schema's guarantees are only as strong as what the database is willing to enforce, and that real data rarely matches the shape its schema implies.

The design decision that required the most deliberation was the relationship itself. Because the original application stored trips and users with no link between them, a faithful migration would have produced two unrelated tables and no foreign keys whatsoever, which would have undercut much of the reason for moving to a relational database in the first place. Introducing the bookings table resolved this, since it gave the schema a genuine relationship to normalize and a place for referential integrity to operate, and it had the additional benefit of filling a gap that the original booking application had left open from the start.

The final lesson came from preserving the application's interface. The goal across the entire enhancement sequence has been to keep the API contract stable so that the Angular frontend continues to work without modification, and the database swap was meant to be no exception. Verifying that claim, however, required running the whole stack rather than testing the API on its own, and doing so revealed gaps that the isolated tests had not, including a mismatch in how the login request was shaped and in how the authentication token was returned to the client. Resolving those so that the unchanged frontend would work against the new database taught me that a preserved contract is proven only by exercising the actual client, and not by trusting that the responses appear correct when viewed in isolation.
