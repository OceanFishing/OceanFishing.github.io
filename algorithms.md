## Artifact Description

The artifact selected for this enhancement is the Travlr Getaways full-stack web application, originally developed during CS 465 in early 2026. By the time of this enhancement, its backend had already been rewritten from Node.js and Express to Python and FastAPI as part of the first enhancement, so the specific component addressed here is the trip listing endpoint, GET /api/trips, within that FastAPI backend. In its original form, that endpoint retrieved the entire trips collection unconditionally and returned every document to the caller, regardless of what the caller actually needed. Although that behavior was functional, it performed no filtering, no ordering, and no limiting of results of any kind, which left the endpoint with no reasoning about how or where its data should be processed.

## Justification for Inclusion

This artifact was selected for the algorithms and data structures category because the question of where filtering, sorting, and controlled result limiting should occur is a clear and self-contained example of the kind of trade-off that the category is meant to demonstrate. The enhancement introduced five optional query parameters, namely a minimum price, a maximum price, a sort field, a sort direction, and a result limit, and the central design decision was where those operations should be applied. Rather than retrieving the full collection into application memory and then narrowing it in Python, I translated the parameters into a single database query that performs the filtering, sorting, and result limiting at the database layer itself.

The specific component that showcases skills in this category is the deliberate decision to push computational work to the part of the system best suited for it, since the database engine can apply indexes and optimized sort routines and return only the documents the caller actually needs, whereas the naive alternative would transfer the entire collection across the network and hold it in memory regardless of how few results were ultimately relevant. Although that difference is negligible against a handful of trips, it scales poorly as the collection grows, and reasoning explicitly about that scaling behavior, rather than defaulting to whichever approach is easiest to write, is the substance of the enhancement. The enhancement also corrected a data representation problem that I had flagged during the original code review, in which the price field had been stored as a string rather than a number. Because comparison operators evaluate strings character by character rather than numerically, a value of nine would be treated as greater than one thousand, so I converted the stored values to numbers before the range filter could be trusted, which made clear that the correctness of an algorithm depends as much on the representation of its data as on the logic itself.

## Course Outcome Alignment

This enhancement was planned to address Course Outcome 3: design and evaluate computing solutions that solve a given problem using algorithmic principles and computer science practices and standards appropriate to its solution, while managing the trade-offs involved in design choices. The outcome has been met, since the central question throughout was the trade-off between processing data in application code and delegating it to the database, and that decision was made deliberately rather than by default. The combined query also demonstrates an understanding of how the operations compose, because the database applies the sort before the limit, which guarantees that a bounded request returns the correct subset, such as the least expensive trips within a price range, rather than an arbitrary selection that merely happens to be ordered among itself.

No updates to the outcome-coverage plan are necessary at this stage. The enhancement delivers on its stated objective without scope creep, and the remaining database enhancement, the migration from MongoDB to PostgreSQL, remains aligned with Outcome 5 as planned.

## Reflection on the Enhancement Process

The process of completing this enhancement was more instructive than its small footprint suggested, since a change that touches only a single function still required reasoning across several distinct layers. The first challenge was the data representation issue described above, which required stepping outside the application code entirely to modify the stored data, and along the way I encountered values that contained leading whitespace, which caused the conversion to fail until those values were trimmed. This was a useful reminder that real data is rarely as clean as the schema implies.

The second challenge was understanding how the database driver actually executes a query, because it was not initially obvious that the call to retrieve trips does not fetch anything on its own and instead returns a cursor that defers execution. The filtering, sorting, and limiting are assembled onto that cursor and run together as a single query only when the results are finally requested, which is what keeps the work at the database layer rather than in application code. Finally, I revisited the way the endpoint reports an empty result. In the earlier version, an empty list returned a not-found status, which was a defensible choice while the endpoint always returned the entire collection. Once filtering was introduced, however, an empty result became an expected outcome of a valid query rather than a sign of a missing resource, so I changed the endpoint to return an empty list with a success status instead. Taken together, these challenges reinforced that even a small and well-scoped enhancement can demand careful reasoning across the stored data, the query execution model, and the meaning of the response itself.
