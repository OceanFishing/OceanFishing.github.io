This ePortfolio presents my CS 499 capstone work. The self-assessment below introduces my background, strengths, and goals, and the sections that follow it contain the original artifact, three enhancements with their narratives, and the code review.

Completing the Computer Science program at Southern New Hampshire University has given me both a clearer sense of the work I want to do and a more honest measure of what I can actually do well. I came into the program after transferring from the University of Florida, and over roughly two years I moved from writing code that merely produced the expected output toward understanding why a given design holds up or fails under pressure. My professional interests have settled on backend engineering and database work, the parts of a system concerned with how data is stored, protected, and served reliably rather than how it is presented, although the current market has convinced me that versatility across the stack is worth more than a narrow specialization. Building this ePortfolio forced me to consolidate that growth into a single defensible demonstration rather than a list of finished courses, and the process clarified which of my strengths are genuinely employable and which still need deliberate work.

Two of the strengths I most want to highlight have less to do with writing code in isolation than with working on code that other people share. Across the program I spent meaningful time reviewing collaborative codebases rather than only my own, which taught me to read unfamiliar code carefully, trace intent through someone else's structure, and raise concerns in a way that improves the work without stalling it. That experience carried directly into the capstone, where a reviewer with professional software experience examined my enhancements and pushed me to justify decisions I had been making on instinct. Closely related is communicating with stakeholders, a skill the program reinforced through the program documentation I produced alongside working software, and most visibly through the code review I recorded for this capstone, where the audience was a peer or manager rather than a grader and the task was to explain not only what the existing code did but why each planned change was worth making. Writing the enhancement narratives extended that same discipline to a written register, since each had to make an engineering decision legible to a reader who had not seen the code.

My grasp of data structures and algorithms is strongest where it matters most for the work I want to do, which is reasoning about the cost of an operation rather than reproducing textbook implementations from memory. I am comfortable thinking about how an approach behaves as the amount of data grows, and about where in a system a given piece of work should happen so that it scales rather than degrades. That reasoning is what led me, later in this portfolio, to move filtering and sorting into the database layer instead of pulling an entire collection into application memory and narrowing it there.

Software engineering, as I now understand it, is less about any single language than about discipline: validating inputs, separating concerns, testing against realistic conditions, and writing code another developer can safely change. My clearest area of database strength is relational design and SQL, including building normalized relational database architecture from a set of requirements, defining the keys, foreign-key relationships, and constraints that hold a schema together, and writing the SQL that retrieves and filters that data efficiently. That foundation is the part of the program I expect to use most directly, since backend and database work is where I want to build my career, and it is the grounding that the original MEAN-stack project, built around a document store, never required me to exercise.

Security is the area I am most deliberately still developing, and the program shifted me from treating it as a final checklist toward treating it as a design constraint present from the first line of code. In practice this shows up in habits rather than in any single dramatic feature: validating and constraining input rather than trusting it, treating user-supplied values as data rather than as code so they cannot alter the queries they reach, hashing credentials rather than storing them in a recoverable form, and pushing guarantees about data into the database itself where a careless code path cannot quietly bypass them. I would not yet describe security as my strongest area, but I have learned to read a piece of code by first asking how it could be misused, and that instinct is one I intend to keep sharpening.

The artifacts that follow this introduction are not three separate projects but three enhancements to a single application, the Travlr Getaways travel booking platform I first built in CS 465, and presenting them together is deliberate. Choosing one artifact and improving it along three different axes shows depth rather than breadth, since each enhancement had to respect the constraints the previous one established. The first enhancement rewrote the entire backend from Node.js and Express to Python and FastAPI while holding the public API contract fixed, so the existing Angular frontend kept working without a single change, which is the software engineering and design category. The second reached into one endpoint and added server-side filtering, sorting, and result limiting, framed around the deliberate trade-off of where computational work should occur, which is the algorithms and data structures category. The third replaced MongoDB with a normalized PostgreSQL schema, introduced a bookings table that finally expressed the relationship the original application had only implied, and moved responsibility for data integrity out of the application and into the database through typed columns and constraints, which is the databases category and the one most aligned with the security mindset the program emphasizes. Read in sequence, the three artifacts trace a single system being made easier to reason about, more efficient at its boundary, and safer at its foundation, and together they are the clearest evidence I can offer of the range this program has given me.
